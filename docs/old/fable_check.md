현황을 다 확인했습니다. 결론부터 말하면 — 그 포지셔닝("개발자 메모리 레이어") 자체는 이 프로젝트에 잘 맞는데, 제안된 기능 우선순위(1→4순위)는 이미 구현된 것과 거의 역순입니다. 그래서 "이 순서로 만들자"가 아니라 "무엇이 진짜 새 것인지"를 걸러내는 게 논의의 핵심이 되어야 합니다.

1. 제안된 우선순위 vs 실제 구현 현황
CLI에 이미 44개 커맨드가 있고, 대조하면 이렇게 됩니다.

제안 우선순위	현황
1순위 세션 연속성	절반 구현. capture-session(세션 요약)은 있음. 없는 건 "다음 세션에서 그걸 자동으로 읽는 쪽" — 그리고 그게 정확히 vault-consumption-plan.md의 get_briefing/search_vault/record_note
2순위 학습 회수	거의 전무. suggest-knowledge가 개념 노트 후보를 만들지만, "AI가 대신 처리해서 내가 이해 못 한 부분"을 추적하는 기능은 44개 커맨드 어디에도 없음
3순위 지식 승격	대부분 완성. promote 파이프라인, write-blog, portfolio-draft, interview-questions, resume, suggest-career-bullets 전부 존재
4순위 자동화 ("나중에")	이미 완성. Telegram 봇, post-commit 훅, nightly-distill, push-digest, related 전부 존재
즉 그 분석은 이 프로젝트를 그린필드로 놓고 짠 로드맵인데, 실제로는 3·4순위가 끝나 있고 1순위의 남은 절반이 지금 열어두신 계획 문서 그 자체입니다. 따라서 vault-consumption-plan의 Phase 1~2(vault_tools + MCP)는 방향 전환과 무관하게 그대로 최우선입니다. 오히려 새 포지셔닝이 그 계획의 근거를 강화해요 — "세션 연속성"은 capture가 아니라 consumption이 없어서 끊기는 것이고, 문서의 "쓰기 전용 메모리" 진단이 정확히 그 얘기니까요.

2. 그 분석이 전제하는 것들 중 점검할 부분
상의니까, 제가 보기에 그 글에서 검증 없이 넘어간 가정 세 가지를 짚겠습니다.

① "세션 연속성"은 차별점으로서 유통기한이 짧아지고 있습니다. Claude Code 자체가 이제 세션 간 자동 메모리, /resume, 컨텍스트 요약-이월을 네이티브로 제공합니다(지금 이 세션도 프로젝트별 메모리 디렉터리를 갖고 있어요). Cursor도 유사 기능이 있고요. "다음 세션 시작 프롬프트 생성" 단독으로는 1~2년 안에 도구에 흡수될 가능성이 높습니다. 반면 도구들이 안 하는 것은 (a) 도구 중립적 저장소(Obsidian) — Claude Code 메모리는 Claude Code 안에 갇힘, (b) 학습 회수, (c) 커리어 자산 파이프라인. 차별점의 무게중심을 1순위가 아니라 2·3순위에 두는 게 구조적으로 안전합니다. 참고로 그 글이 언급한 OpenClaw/Hermes는 제가 실체를 확인할 수 없는 이름이라, 경쟁 구도 판단의 근거로 삼으려면 별도 검증이 필요합니다(원하시면 deep-research로 경쟁 도구 스캔을 돌려볼 수 있습니다).

② 학습 회수는 "생성"이 아니라 "소비"가 병목입니다. 복습 질문 3개를 만들어주는 건 LLM 호출 하나로 끝나는 쉬운 일이에요. 어려운 건 그 질문에 언제, 어디서, 실제로 답하게 만드느냐입니다. 이걸 설계 안 하면 60_Candidates에 미확인 학습 노트가 쌓이는 — 방금 promote 병목을 고치면서 겪은 것과 똑같은 — 쓰기 전용 메모리를 하나 더 만들게 됩니다. 다행히 이 프로젝트엔 이미 소비 채널이 있어요: push-digest가 Telegram으로 daily digest를 보내니까, 복습 질문을 digest에 실어 보내는 것이 새 앱 없이 루프를 닫는 최단 경로입니다.

③ "제품 포지셔닝"과 "포트폴리오 프로젝트"가 섞여 있습니다. 남이 쓸 제품이면 온보딩·범용성이 우선이고, 본인용 + 포트폴리오면 "내 문제를 실제로 해결하는 살아있는 루프 + 그걸 설명하는 서사"가 우선입니다. 이건 기능 우선순위를 실제로 바꾸는 갈림길이라 사람이 정해야 할 판단인데, 지금까지의 설계(로컬 stdio, 개인 vault 전제, 이력서/면접 커맨드)를 보면 사실상 후자로 이미 기울어 있고, 저도 그게 맞다고 봅니다. 그렇다면 "포지셔닝 변경"의 실체는 코드가 아니라 README/docs의 서사 재작성 + 학습 회수 레이어 하나 추가입니다.

3. 제 제안 — 기존 계획에 한 레이어만 얹기
로드맵을 갈아엎을 필요 없이 이렇게 가는 걸 추천합니다.

vault-consumption-plan Phase 1~2 그대로 진행 (feat/vault-mcp). 이게 새 포지셔닝의 1순위 "세션 연속성"의 본체입니다. record_note가 decision-log, get_briefing이 next-session-prompt 역할.
학습 회수를 capture-session 스키마 확장으로 시작. 새 커맨드 6개(quiz-me, unknowns-list...)를 만들기 전에, CLAUDE.md의 capture-session 작성 기준에 "학습 회수" 섹션 하나를 추가하는 것으로 충분합니다 — "AI가 대신 처리한 부분 / 내가 아직 애매한 개념 / 복습 질문 2~3개". 비용은 문서 수정뿐인데, 매 세션마다 데이터가 쌓이기 시작합니다. 커맨드 표면적을 늘리는 건 이 데이터가 실제로 유용하다고 확인된 다음에 해도 늦지 않아요.
복습 질문을 nightly-distill → push-digest 경로에 합류시켜 소비 루프를 닫기. "어제 세션의 미해결 개념 1개 + 복습 질문"이 아침 Telegram digest에 실리면, 별도 습관 없이 학습 회수가 돌아갑니다. ②의 함정을 피하는 핵심.
서사 재작성은 마지막에. MCP + 학습 회수가 실제로 돌아간 뒤에 README와 블로그 글("AI 코딩 시대의 개발자 메모리 레이어")을 쓰면, 포트폴리오 서사가 구현 사실에 기반하게 됩니다. 참고로 이 재포지셔닝 논의 자체가 60_Candidates/Decisions/ 후보로 남길 가치가 있는 결정이기도 합니다.
만들지 말 것도 명확합니다: 그 글의 explain-changes(Claude Code가 이미 함), 범용 비서화, 그리고 지금 시점의 임베딩 검색(계획대로 Phase 4 조건부 유지).

정리하면 — 그 포지셔닝은 채택하되, 받아들일 것은 "우선순위 목록"이 아니라 "학습 회수라는 빠진 레이어"와 "서사" 두 가지이고, 실행 순서는 기존 계획 문서가 이미 맞게 잡아놨다는 게 제 판단입니다. 어느 지점부터 파고들지(Phase 1 착수, capture-session 스키마 확장, 아니면 경쟁 도구 검증 리서치) 정해주시면 이어가겠습니다.
